# 三阶段文档工作流

本项目的文档体系采用三阶段递进工作流，将"需求分析"、"产品设计"与"开发实现"拆解为三个层次，每层有明确的输入、输出和边界，确保 Agent 在各自职责范围内工作，避免在"思考价值"与"思考实现"之间混淆。

---

## 阶段边界与核心差异

| 维度 | BRD（需求分析） | PRD（产品设计） | SRD（开发需求） |
|------|----------------|----------------|----------------|
| **核心问题** | 为什么做？ | 做什么？ | 怎么做？ |
| **视角** | 用户视角 + 商业视角 | 产品视角 + 交互视角 | 技术视角 + 工程视角 |
| **关注重点** | 需求价值、用户痛点、业务流程合理性、可实现性评估 | 功能模块划分、交互流程、信息架构、原型 | 接口定义、数据模型、状态机、测试规格、性能指标 |
| **输出性质** | 决策依据（做不做） | 设计蓝图（长什么样） | 施工图纸（如何编码） |
| **评审角色** | 业务方、用户代表 | 产品经理、设计师、核心用户 | 技术负责人、Coding Agent |
| **变更成本** | 高（涉及需求方向） | 中（原型可反复修改） | 低（开发中可微调） |

### BRD：需求分析 —— "为什么做"

BRD 的核心产出是**"需求是否值得做、用户是否真需要"**，而不是"功能怎么摆"。它站在用户视角回答：

- 用户现在怎么做的？（现状 AS-IS）
- 用户卡在哪里？（痛点）
- 做了之后用户会怎么变？（效果）
- 不做会怎样？（机会成本）

BRD 包含**用户故事**和**业务流程（业务视角）**。这里的流程是**用户旅程/业务流**，关注"用户为了达成目标需要经历哪些步骤，每一步是否合理"，不关心具体界面和点击位置。

BRD 还负责**需求边界澄清**（In-Scope / Out-of-Scope），防止后续阶段范围蔓延。

### PRD：产品设计 —— "做什么"

PRD 在 BRD 已确认的需求范围内开展设计，关心的是**功能模块的划分、每个子功能之间如何衔接、组装形成完整的功能模块**。

PRD 将 BRD 中的业务流细化为**交互流程**，关注"用户在什么页面、做什么操作、系统如何响应、页面如何跳转"。

PRD 还涉及**原型制作**，文档里的每个功能点都应该能在原型中找到对应。PRD 与原型一起配合，向用户解释需求是如何被满足的。

**关键原则**：同一个 BRD 可以对应多个 PRD 方案。例如 BRD 确认"用户需要快速找到历史订单"，PRD 可以产出：
- 方案 A：顶部搜索框 + 筛选器（高效型）
- 方案 B：时间轴 + 卡片流（浏览型）
- 方案 C：AI 对话式查询（创新型）

### SRD：开发需求 —— "怎么做"

SRD 在 PRD 已确认的设计方案基础上，定义**功能真实被实现的方案**。PRD 是设计蓝图，SRD 是施工图。

两者的典型差异：

| 维度 | PRD（蓝图） | SRD（施工图） |
|------|------------|--------------|
| **接口定义** | 说明"需要调用支付接口，核心参数是订单号和金额" | 定义接口路径 `/api/v1/pay`，参数命名 `order_id`（String, 32位）、`amount`（Number, 分单位）、校验规则、错误码枚举 |
| **数据模型** | 说明"订单需要记录支付状态、支付时间" | 定义 `orders` 表结构，字段类型、索引、外键、默认值 |
| **状态流转** | 画出"待支付 → 支付中 → 已支付 → 已发货"的业务状态机 | 定义状态字段 `status`（Enum）、转换触发条件、幂等性处理 |
| **触发逻辑** | 说明"用户提交订单后触发支付" | 定义事件监听机制、消息队列 Topic、异步回调处理、超时补偿策略 |

SRD 定义了除代码实现层面之上的所有技术设计。Coding Agent 拿到 SRD 后，可以依据其中的 spec 开展施工，施工效果以 SRD 中的验收标准是否得到满足为依据。

## 阶段一：需求分析阶段

**目标**：将混沌的初始想法转化为结构化、可决策的商业价值说明。

### 阶段一前置：RIL（需求接收日志）

RIL（Requirement Intake Log，需求接收日志）是可选的轻量前置步骤，用于在正式分析前忠实记录原始需求线索。它明确区分"提出"与"分析"的边界，允许信息不全、允许粗糙，不追求一次完美。

- **编号格式**：`RIL-YYYY-MM-NNN`
- **使用时机**：收到零散需求反馈、用户工单、销售/CSM 口头反馈等尚未经过梳理的原始线索时
- **处理方式**：多个 RIL 可汇总分析，提炼后进入 BRD；单个 RIL 信息足够充分时也可直接转入 BRD
- **模板参考**：[`knowledge/00-meta/RIL-example.md`](knowledge/00-meta/RIL-example.md)

**输入文档**：
- 原始需求素材：用户反馈、业务方访谈记录、客服工单、竞品分析、市场趋势
- 背景材料：产品愿景、目标用户画像、业务线数据（DAU、转化率等）
- 约束条件：技术限制、资源时间限制、合规要求、过往同类需求的成败记录

**输出文档**：
- **《需求分析报告》（BRD）**
  - 推荐格式：Markdown（方便后续 Agent 快速读取）
  - 核心内容：
    - 需求本质与目标（5W2H、North Star Metric）
    - 用户故事（含验收标准 AC）
    - 现状与痛点分析（AS-IS）
    - 目标业务流程（TO-BE，业务视角，只关心角色和步骤，不关心界面）
    - 需求范围与边界（In-Scope / Out-of-Scope）
    - 商业价值分析（ICE/RICE 评分、ROI 预估）
    - 需求可实现性评估（技术可行性、资源可行性、风险评估）
    - 决策建议（推进 / 暂缓 / 拒绝 / 需补充调研）

**Agent 工作方式**：使用 `/requirement-analyzer` skill，通过 5W1H 提问挖掘真实需求。

## 阶段二：产品原型设计阶段

**目标**：将需求报告转化为具体的可视化交互方案，并产出带有设计意图的交互说明。

**输入文档**：
- 《需求分析报告》（阶段一产出）
- 全局设计规范：UI 组件库、品牌指南、Design System
- 技术与前端约束：当前前端技术栈能力
- 参考物：历史类似原型或竞品界面

**输出文档**：
- **《产品设计文档》（PRD）**
  - 推荐格式：Markdown
  - 核心内容：
    - 设计目标（继承 BRD 的目标和验收标准）
    - 需求范围确认（复制 BRD 的 Scope，确保设计不越界）
    - 多版本方案对比与评估矩阵（MVP版 / 完整版 / 创新版）
    - 功能架构设计（模块划分、模块间关系）
    - 用户旅程与交互流程（基于 BRD 的 TO-BE 流程细化为交互层面的流程）
    - 信息架构（页面结构、导航设计）
    - 核心功能详细设计（业务逻辑合理性、用户交互逻辑、接口调用逻辑——只列核心参数和用途）
    - 异常处理逻辑（功能层面的异常，如网络错误降级、无数据状态）
    - 交互规范与视觉规范引用
    - 待确认问题（评审时需讨论的设计难点）

- **可交互 Prototype**
  - 推荐格式：可运行的轻量级前端网页（本项目原型位于 `/prototype` 目录）
  - 核心内容：完整主流程、带真实交互逻辑的操作反馈
  - PRD 中描述的每个功能点都要能在原型中找到对应

## 阶段三：开发需求编写阶段

**目标**：将产品方案转译为逻辑严密、毫无歧义的"工程图纸"，直接驱动 Coding Agent 编写代码。

**输入文档**：
- 《需求分析报告》（获取业务背景和目标）
- 《产品设计文档》与 Prototype 文件（获取 UI 和交互规则）
- 技术架构文档（获取系统边界、已有 API 规范）

**输出文档**：
- **《开发需求文档》（SRD）**
  - 推荐格式：Markdown + Mermaid.js（绘图）+ JSON Schema（数据）
  - 核心内容：
    - 需求目标回顾（复制 BRD 的业务目标，确保开发团队理解"为什么做"）
    - 方案概述（说明设计方案如何解决 BRD 中定义的痛点）
    - 功能清单（模块、功能点、优先级、对应 PRD 章节）
    - 功能详细规格（IPO 定义：Input-Process-Output）
    - 状态机定义（状态、触发事件、条件、下一状态、副作用）
    - 接口定义（API Specification：路径、方法、请求参数、响应结构、错误码）
    - 数据模型（表结构、字段类型、索引、外键）
    - 时序图（用户 → 前端 → API → Service → DB → 第三方）
    - 异常处理（异常场景、触发条件、系统行为、用户反馈、日志要求）
    - 非功能需求（性能、安全、兼容性、可观测性）
    - 测试规格（测试策略、重点用例、回归范围）
    - 上线与验收标准（功能验收 Checklist、数据埋点、灰度策略、回滚方案）

- **测试用例（TC）**、**上线检查清单（CKL）**、**用户手册（MAN）** 作为 SRD 的配套文档产出。

## 文档间的引用关系

```
RIL [需求接收日志]
└── 原始线索 → 汇总分析后输入 BRD

BRD [需求分析]
├── 目标 → 被 PRD 继承
├── 用户故事 → 被 PRD 的功能设计覆盖
├── 业务流程（TO-BE）→ 被 PRD 的交互流程细化
├── 范围（Scope）→ 约束 PRD 的设计边界
└── 商业价值 → 被 SRD 的方案概述引用

PRD [产品设计]
├── 功能架构 → 被 SRD 的功能清单拆解
├── 交互流程 → 被 SRD 的时序图技术化
├── 接口逻辑（PRD层）→ 被 SRD 的 API 定义精确化
├── 数据逻辑（PRD层）→ 被 SRD 的数据模型精确化
└── 原型 → 被 SRD 的验收标准验证

SRD [开发需求]
├── 继承 BRD 的"为什么"
├── 继承 PRD 的"做什么"
└── 定义"怎么做 + 怎么测"
```

**变更传播原则**：
- BRD 变更 → 必须同步审查 PRD 和 SRD 是否受影响
- PRD 变更 → 必须同步更新 SRD 中的技术规格
- SRD 变更 → 通常只影响实现细节，不影响 BRD 和 PRD 基线
